### Tooling discovery (preview-only, NOT an app bug)
In the embedded preview browser, CSS `animationend` never fires, so react-native-web `Modal` fade-outs never complete — dismissed modals leave zombie DOM even though React state is correctly `visible=false`. When verifying modal dismissal there, inspect React fiber props or downstream state effects, not DOM presence. Real browsers and devices are unaffected (and native iOS uses native modals entirely).

### Follow-up — web onboarding/balances "reset" report (investigated, no bug)
User reported: onboarding restarts from the beginning on every browser refresh, and starting balances entered during onboarding "are not taken into consideration." Asked user to confirm platform: web browser.

Root cause confirmed by direct testing, not just re-reading the existing comment: `db/database.ts` uses `:memory:` SQLite on web, which resets on every page reload — both symptoms (onboarding flag gone, balances gone) are the same reset, not two separate bugs. Verified live:
- With a single tab, reload → onboarding correctly reappears (fresh empty DB, as designed).
- Within one session (no reload), setting balances (300/150/800) and continuing immediately rendered all three "$X left" lines correctly — the save/apply path itself has no bug.
- Along the way hit a real but unrelated error — `NoModificationAllowedError: createSyncAccessHandle ... Access Handle cannot be created if there is another open Access Handle` — caused by having two preview tabs open to the same origin at once (an OPFS-style lock race). This is a dev-tooling artifact from testing, not something end users hit; closing the stale tab resolved it.

User decision: leave web behavior as-is (documented, expected); do not build web persistence. No code changes made for this item.

### Feature — expanded keyword dictionary + manual keyword management

User asked for a "more powerful dictionary" for Fast input and whether Android
payment-notification reading was feasible. The notification question was answered
as informational-only (native notification-listener service required, no Expo Go
support, manual permission grant, Play Store declared-use policy, fragile per-bank
parsing) — not implemented, logged in [[ChronoBudget]] Next steps. The dictionary
request became three pieces of work, the third added mid-plan-review after the user
asked about letting users add their own words:

1. **Restructured `constants/keywordMap.ts` into a per-language registry**, mirroring
   `lib/i18n.ts`'s `BUNDLES` pattern: `constants/keywords/en.ts` (`EN_KEYWORDS`) +
   `constants/keywords/index.ts` (`KEYWORD_MAPS`/`getKeywordMap()`); the old
   `constants/keywordMap.ts` path is now a thin re-export so `lib/detectCategory.ts`
   needed no changes.
2. **Grew the seed dictionary from 77 to ~350 entries** (hand-curated, no external
   dictionary API — offline-first constraint), covering more grocery/retail chains,
   transport/rideshare brands, delivery apps, streaming/subscription services, etc.
   Fixed one silent bug while curating: a duplicate `subway` key (transit vs. the
   sandwich chain) would have had the later object-literal entry silently win;
   removed the dining duplicate. Also had to quote `'401k'` (invalid as a bare
   identifier key) and lowercase `councilTax` → `counciltax` (matching lookup is
   always lowercased, so a mixed-case key could never match).
3. **New "My Keywords" manual management UI** (`components/KeywordsModal.tsx`,
   opened from a new row in `SettingsModal.tsx`) — add/edit/delete keyword →
   category/subcategory mappings directly, reusing the existing `keyword_learn`
   table and `learnKeyword()`/`fetchLearnedKeywords()` (no new storage) plus one new
   DB function, `deleteLearnedKeyword()`. Chip UI copied from
   `EditTransactionModal.tsx`'s category/subcategory pattern.

Vault decision note updated: [[smart-input-classifier]] (new "Future work" section
documents fuzzy/stemming matching as deliberately deferred, and the active-locale
keyword-map swap as a small follow-up).
