### Feature — automated testing plan and initial suite

User asked to start an automated testing plan, with tests that must pass on each build. Codebase had zero test infrastructure (no jest, no test files, no CI, no lint script — confirmed via direct `package.json`/`node_modules` inspection, not assumption). Scoped with the user to: unit tests (Jest) + component tests (RNTL), run locally via `npm test` before builds — no CI, no E2E for this pass.

Installed via `npx expo install jest-expo jest @testing-library/react-native @types/jest --dev` (SDK-56-matched versions). Added `test`/`test:watch` scripts and a `"jest": { "preset": "jest-expo" }` block to `package.json`.

**Hit and fixed two setup issues:**
1. First attempt hand-copied a `transformIgnorePatterns` override from older Expo/Jest docs — broke immediately on `expo-modules-core` (`Cannot use import statement outside a module`), because it overwrote jest-expo's own better-maintained default pattern. Fix: removed the override, let the preset's default handle it.
2. `@testing-library/react-native` v14's `render()` is async; calling it without `await` in `BentoCard.test.tsx` failed every query with `` `render` function has not been called `` (the internal `screen` singleton isn't bound until the returned promise resolves). Fixed by awaiting `render()` in every test.

Also added `"types": ["jest"]` to `tsconfig.json` so `npx tsc --noEmit` recognizes `describe`/`it`/`expect`/`jest.*` (this project has no separate test tsconfig).

**New test files (50 tests, all passing):**

- `lib/__tests__/format.test.ts` — `formatCurrency`/`formatCompactCurrency`/`formatNumber`/`formatDate`, including the `Intl`-throws fallback path (mocked `Intl.NumberFormat`/`Intl.DateTimeFormat`), seeding `useBudgetStore` state directly since these functions read `getState()`.
- `store/__tests__/useBudgetStore.test.ts` — simple setters (`triggerRefresh`, `setCategoryTotals`, `setRecentTransactions`, `setLimits`, `setBalances`) plus async actions (`loadLearnedKeywords`, `loadRecurring`, `loadLocale`, `setCountry`) with `db/database.ts` mocked via `jest.mock`.
- `components/__tests__/BentoCard.test.tsx` — formatted amount, balance/remaining line (positive/negative/absent), limit progress bar color tiers, OVER badge shown/hidden.

Confirmed by direct test that `expo-sqlite`'s native module isn't available under jest-expo's default test environment (`TypeError: _ExpoSQLite.default.NativeDatabase is not a constructor` when calling `getDb()`). Building a working SQLite mock/harness was judged disproportionate infra for this pass per CLAUDE.md's "avoid premature complexity" guidance — logged as a known gap rather than forced. `db/database.ts` (migrations, CRUD, `processRecurring`) remains untested.

### Feature — Phase 2: closed the three testing follow-ups (DB coverage, remaining components, CI)

Same day, follow-up work: user asked to close the three gaps flagged as next steps after Phase 1 — DB coverage, component tests for the remaining DB-writing components, and CI.

**1. DB coverage.** Rather than re-accept the Phase 1 gap, mocked the `expo-sqlite` module itself with a real in-memory SQL engine so `db/database.ts` runs unmodified against genuine SQL:
- New root-level `__mocks__/expo-sqlite.ts` (Jest's manual-mock convention for node_modules packages requires `<rootDir>/__mocks__/`, not nested under `db/`), wrapping **`sql.js`** — pure-JS/WASM SQLite, no native build step.
- Hit a real pitfall: sql.js's default WASM build throws an unhelpful empty `Error` from `new SQL.Database()` under Jest. Root cause: a known Jest/WebAssembly incompatibility — Jest runs each test file in its own `vm` context, and V8's WASM instances don't behave correctly across realms. Fix: switched to `sql.js/dist/sql-asm.js`, the pure-JS build with no WASM, which has no such issue.
- New `db/__tests__/database.test.ts` (9 tests): migration ladder reaches `user_version = 6` with all 6 tables; a deliberate `CHECK constraint failed` proves the mock runs real SQL, not a stub; settings CRUD; `setLimit`/`setBalance` delete-on-`<=0` convention; transactions CRUD + totals; keyword upsert; `processRecurring` catch-up (mocked `Date.now()`, verified 4 missed monthly occurrences post correctly and `next_run` advances using the real `advance()`); `fetchMonthlyTotals` bucketing. `getDb()`'s module-level memoization is worked around with `jest.resetModules()` + re-require per test for isolation.

**2. Component tests for the remaining DB-writing components** — same `jest.mock('../../db/database')` pattern proven in Phase 1's store tests:
- `components/__tests__/ExpenseInput.test.tsx` (6 tests): fast-mode auto-detection preview via the real `detectCategory`, mode-toggle persistence, amount validation, submit + conditional keyword-learning.
- `components/__tests__/EditTransactionModal.test.tsx` (5 tests): prefill (preset vs. custom subcategory), validation, save, delete.
- `components/__tests__/RecurringModal.test.tsx` (6 tests): empty/seeded list, add validation, add-flow save sequencing (`insertRecurring` → `processRecurring` → `loadRecurring` → `triggerRefresh`), edit prefill + `updateRecurring`, delete.
- `components/__tests__/KeywordsModal.test.tsx` (7 tests): empty/seeded list, add validation (word/subcategory), save, edit prefill, delete.
- **Pitfall hit and fixed:** RNTL v14's `fireEvent.*` is *also* async (not just `render()`, the Phase 1 lesson). An un-awaited `fireEvent.changeText(...)` followed immediately by `fireEvent.press(...)` let the press fire before the text-change state update flushed, so saves went through with stale field values — surfaced as wrong call arguments plus React "overlapping act()" warnings. Every `fireEvent.*` call now `await`s.
- Added one-line `testID`s to three icon-only delete buttons (`EditTransactionModal`, `RecurringModal` per-rule, `KeywordsModal` per-keyword) since they had no text/accessible name to query by — a minimal, no-behavior-change addition.

### Feature — Phase 3: closed the last 2 testing gaps (migration idempotency, E2E)

User asked to close the final two gaps from `testing-strategy.md`. For E2E, chose **web E2E via Playwright** over native Detox/Maestro (no simulator/emulator/macOS-runner infra needed; Detox/Maestro would need a dev build plus a running simulator/emulator, and this Windows machine has no local Android emulator set up — the macOS VM already used for iOS testing would only cover iOS, not Android).

**1. Migration idempotency.** `openAndMigrate()` exported from `db/database.ts` (matches the existing `getDb`/`initDb` export pattern, no behavior change). The bigger piece: `__mocks__/expo-sqlite.ts`'s `openDatabaseAsync` previously created a **fresh, empty** `sql.js` database on every call regardless of name — meaning calling `openAndMigrate()` twice couldn't test anything real. Changed it to memoize by name in a `Map`, so repeat opens of `'chronobudget.db'` return the same underlying data — mirroring how real native SQLite reuses the same on-disk file across app relaunches. This is also a correctness improvement to the mock itself, not just a test hook. New test inserts data, calls `openAndMigrate()` again against the same name, asserts no error, `user_version` still 6, and the data survived (proving the `CREATE TABLE IF NOT EXISTS` / `DROP TABLE IF EXISTS` guards in the migration SQL actually work as intended).

**2. Web E2E via Playwright.** New `e2e/` directory: `helpers.ts` (`completeOnboarding()` — web's in-memory DB resets every page load, so each spec gets a guaranteed-fresh onboarding state), `onboarding.spec.ts`, `transactions.spec.ts` (add/edit/delete a transaction from the dashboard), `navigation.spec.ts` (transaction shows up in History; Trends renders with zero console errors). `playwright.config.ts` spins up `npx expo start --web --port 8081` via Playwright's `webServer` option.

Hit three setup pitfalls:
1. Playwright's `getByText()` substring-matches by default — `getByText('Continue')` ambiguously matched both the real button and unrelated copy containing the word "Continue". Fixed with `{ exact: true }`.
2. Playwright has no RNTL-style `getByDisplayValue()` — added a `testID="edit-amount-input"` to `EditTransactionModal`'s amount field (same pattern as the existing delete-button testIDs) and asserted via `toHaveValue()`.
3. Jest's default `testMatch` also picked up `e2e/*.spec.ts`, and Playwright's `test()` throws when run inside Jest. Fixed with `"testPathIgnorePatterns": ["/node_modules/", "<rootDir>/e2e/"]` in the `jest` config.

Also hit a real flake during the transactions spec: reopening the edit modal immediately after Save occasionally raced the modal's close animation, causing a transient duplicate "Dining" text match. Fixed by asserting the amount input is hidden (modal fully closed) before reopening, rather than adding an arbitrary wait.

**CI:** added a parallel `e2e` job to `.github/workflows/ci.yml` (`npm ci && npx playwright install --with-deps chromium && npm run test:e2e`).
